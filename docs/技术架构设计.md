# 租房微信小程序技术架构设计

## 1. 设计目标

- 建立清晰的前后端分层结构
- 统一页面、服务层、云函数、数据库之间的调用边界
- 彻底消除“多个账号共享同一个 `_openid` 导致串号”的旧问题
- 为微信登录、手机号登录、显式微信绑定、会话鉴权提供稳定基础

## 2. 技术选型

- 前端框架：微信小程序原生开发
- 后端能力：微信云开发
- 后端组成：云函数 + 云数据库 + 云存储
- 语言：JavaScript
- 异步规范：统一使用 `async/await`

## 3. 总体架构

```text
页面层 pages / package-*
        ↓
服务层 services/*
        ↓
云调用封装 services/cloud/*
        ↓
云函数 cloudfunctions/*
        ↓
云数据库 / 云存储
```

认证链路补充：

```text
客户端 accessToken + userInfo
        ↓
services/cloud/call.js 自动注入 auth.accessToken
        ↓
受保护云函数 resolveCurrentUser()
        ↓
user_sessions -> userId -> users
```

设计原则：

- 页面层不直接写 `wx.cloud.callFunction`
- 页面层不直接操作数据库
- 所有服务调用统一经过 `services/cloud/call.js`
- 云存储上传统一经过 `services/cloud/upload.js`
- 当前用户识别统一走 session，不再依赖 `users._openid`

## 5. 前端分层设计

### 5.1 页面层

职责：

- 维护页面状态
- 响应用户交互
- 调用服务层 / store
- 渲染 UI

约束：

- 不直接调用云函数
- 不直接拼装底层数据库对象
- 不自行维护“当前用户是否合法”的判定逻辑

### 5.2 服务层

职责：

- 统一调用云函数
- 统一封装参数与返回值
- 统一处理异常
- 隔离页面与后端实现细节

主要文件：

- `auth.service.js`
- `user.service.js`
- `house.service.js`
- `favorite.service.js`
- `history.service.js`
- `chat.service.js`
- `map.service.js`
- `bootstrap.service.js`

### 5.3 状态层

职责：

- 管理 `accessToken`
- 管理 `userInfo`
- 管理登录态恢复
- 在 token 失效时清理本地会话

主要文件：

- `app.store.js`
- `user.store.js`

### 5.4 工具层

职责：

- 提供纯工具方法
- 封装 storage / auth / format / validate / logger

说明：

- `utils/storage.js` 负责 `accessToken` 与 `userInfo` 的本地持久化
- `utils/auth.js` 负责登录态判断和路由守卫

## 7. 统一云调用设计

### 7.1 统一调用入口

文件：

- `services/cloud/call.js`

职责：

- 统一拼装 `name`、`action`、`payload`
- 自动附带 `auth.accessToken`
- 统一处理成功和失败格式
- 统一抛出业务异常

### 7.2 请求格式

```json
{
  "action": "getCurrentUser",
  "payload": {},
  "auth": {
    "accessToken": "client_token"
  }
}
```

其中：

- 公共接口可不带 `auth`
- 受保护接口必须带有效 `accessToken`

## 9. 用户身份与登录态模型

当前统一口径：

- `userId`：业务主键
- `user_identities`：登录身份映射
- `user_sessions`：登录会话
- `accessToken`：客户端会话凭证
- `userInfo`：客户端展示缓存

### 9.1 微信登录

流程：

1. 通过微信运行时拿到 `OPENID`
2. 在 `user_identities(type=wechat_openid)` 中查找已绑定用户
3. 若已绑定，直接创建 session
4. 若未绑定，创建新用户并建立微信身份映射，再创建 session

### 9.2 手机号注册 / 登录

流程：

1. 注册时创建 `users`
2. 创建 `user_identities(type=phone)`
3. 创建 `user_sessions`
4. 返回 `accessToken + userInfo`

重要约束：

- 手机号注册 / 登录不自动绑定微信身份
- 同一个微信身份不能共享给多个业务账号

### 9.3 微信绑定 / 解绑

流程：

1. 用户先通过现有账号登录
2. 服务端拿当前微信 `OPENID`
3. 检查该 `OPENID` 是否已绑定其他用户
4. 未绑定才创建 `user_identities(type=wechat_openid)`
5. 解绑时仅禁用该身份映射，不影响手机号身份

### 9.4 受保护接口鉴权

流程：

1. 客户端请求携带 `accessToken`
2. 云函数读取 `auth.accessToken`
3. 在 `user_sessions` 中校验 session
4. 通过 `userId` 读取 `users`
5. 执行业务逻辑

## 11. 技术约束

- 微信小程序原生开发，不引入 `uni-app`、`Taro`
- 页面禁止直接调用 `wx.cloud.callFunction`
- 页面禁止直接访问数据库
- 所有服务调用统一经由 `services` 层
- 所有受保护接口统一走 token 鉴权
- 敏感字段必须在服务端完成加密、脱敏和权限校验
- 初始化逻辑不得在每次启动时自动写库
- 测试数据能力不得暴露到正式业务页面

## 12. 风险与注意事项

- 若未来导入旧数据，需要将旧 `_openid` 迁移到 `user_identities`
- 历史数据若仍混用 `phone` / `_openid` / `userId`，必须先补齐为 `userId`
- 聊天与通知结构调整时应继续保持 `participantIds / senderId / receiverId` 口径
- 手机号换绑不能复用 `updateProfile`，必须单独设计验证码校验和身份迁移
