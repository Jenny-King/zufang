# 租房微信小程序数据库设计说明

## 1. 文档目标

- 明确当前版本小程序的数据库集合设计
- 统一字段命名、数据关系和鉴权模型
- 彻底规避 `userId`、`phone`、`_openid` 混用导致的身份混乱问题
- 为后续索引优化、身份迁移和业务扩展提供基础

## 2. 设计原则

- 业务关联统一使用 `userId`
- 身份映射与登录会话必须拆表管理
- 页面不直接操作数据库，统一由云函数处理
- 敏感信息不以明文对前端输出
- 当前用户识别统一依赖 session，不依赖 `_openid`

## 3. 用户身份模型

### 3.1 三层结构

- `users`：用户资料主体
- `user_identities`：身份映射
- `user_sessions`：登录会话

### 3.2 字段职责

- `userId`：业务主身份标识，面向收藏、历史、聊天、房源归属等业务关系
- `phone`：用户资料字段，同时通过 `user_identities(type=phone)` 参与登录
- `wechat_openid`：仅存在于 `user_identities(type=wechat_openid)` 中
- `accessToken`：客户端会话凭证，对应 `user_sessions`

### 3.3 约束

- 新数据统一使用 `userId` 作为业务关联字段
- 同一个身份标识只能映射到一个有效用户
- 同一个账号可以拥有多个 session
- `updateProfile` 不允许直接修改手机号

## 4. 集合设计

### 4.1 `users`

用途：

- 存储用户资料、角色、认证状态和密码哈希

字段设计：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | String | 是 | 云数据库文档 ID |
| `userId` | String | 是 | 业务用户 ID，建议唯一 |
| `phone` | String | 否 | 手机号资料字段 |
| `nickName` | String | 是 | 昵称 |
| `avatarUrl` | String | 否 | 头像地址 |
| `role` | String | 是 | `tenant` / `landlord` / `admin` |
| `gender` | Number | 否 | 性别枚举 |
| `province` | String | 否 | 省 |
| `city` | String | 否 | 市 |
| `district` | String | 否 | 区 |
| `wechatId` | String | 否 | 微信号 |
| `verified` | Boolean | 是 | 是否已通过身份审核 |
| `identityStatus` | String | 否 | `unsubmitted` / `pending` / `approved` |
| `identitySubmittedAt` | Date | 否 | 身份资料提交时间 |
| `realName` | String | 否 | 真实姓名 |
| `idCardEncrypted` | String | 否 | 加密后的身份证号 |
| `idCardMasked` | String | 否 | 脱敏身份证号 |
| `passwordHash` | String | 否 | 密码哈希值 |
| `loginType` | String | 否 | 用户初始来源，如 `wx` / `phone` |
| `status` | String | 是 | `active` / `disabled` |
| `createTime` | Date | 是 | 创建时间 |
| `updateTime` | Date | 是 | 更新时间 |

索引建议：

- `userId` 唯一索引
- `phone`
- `role`
- `status`

说明：

- `users` 不再保存 `_openid`
- 不建议再向前端返回明文 `idCard`

### 4.2 `user_identities`

用途：

- 存储“某个登录身份属于哪个用户”

字段设计：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | String | 是 | 推荐由 `type:identifier` 哈希生成 |
| `type` | String | 是 | `phone` / `wechat_openid` |
| `identifier` | String | 是 | 手机号或微信 `OPENID` |
| `userId` | String | 是 | 归属用户 |
| `status` | String | 是 | `active` / `disabled` |
| `createTime` | Date | 是 | 创建时间 |
| `updateTime` | Date | 是 | 更新时间 |

索引建议：

- `_id` 唯一索引
- `userId`
- `type + identifier`

说明：

- 同一条 `identifier` 不能绑定到多个 active 用户
- 微信绑定与解绑只操作本集合

### 4.3 `user_sessions`

用途：

- 存储登录会话

字段设计：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `_id` | String | 是 | 推荐使用 `tokenHash` |
| `tokenHash` | String | 是 | `accessToken` 的哈希值 |
| `userId` | String | 是 | 会话归属用户 |
| `source` | String | 是 | `wechat` / `register` / `phone_code` / `phone_password` 等 |
| `status` | String | 是 | `active` / `revoked` |
| `expireAt` | Date | 是 | 过期时间 |
| `lastUsedTime` | Date | 否 | 最近使用时间 |
| `createTime` | Date | 是 | 创建时间 |
| `updateTime` | Date | 是 | 更新时间 |

索引建议：

- `_id` 唯一索引
- `userId`
- `status`
- `expireAt`

说明：

- 前端不保存 `tokenHash`，只保存明文 `accessToken`
- 服务端只落 `tokenHash`

## 5. 集合关系说明

### 5.1 用户与身份

- 一个 `users.userId` 可以对应多个身份
- `user_identities.userId -> users.userId`

### 5.2 用户与会话

- 一个用户可以拥有多个 session
- `user_sessions.userId -> users.userId`

### 5.3 用户与房源

- 一个房东可以发布多个房源
- `houses.landlordUserId -> users.userId`

### 5.4 用户与收藏 / 历史

- `favorites.userId -> users.userId`
- `history.userId -> users.userId`
- `favorites.houseId -> houses._id`
- `history.houseId -> houses._id`

### 5.5 用户与聊天

- 一个会话关联两个用户
- `conversations.participantIds -> users.userId`
- `chat_messages.conversationId -> conversations.conversationId`

### 5.6 用户与通知

- `messages.userId -> users.userId`

## 6. 数据输出约定

### 6.1 禁止直接返回前端的字段

- `users.passwordHash`
- `users.idCardEncrypted`
- `user_identities.identifier`
- `user_sessions.tokenHash`
- `user_sessions.*`

### 6.2 推荐用户输出结构

```json
{
  "userId": "user_1001",
  "nickName": "张三",
  "avatarUrl": "",
  "phone": "138****0000",
  "role": "tenant",

  "wechatId": "",
  "province": "",
  "city": "",
  "district": "",
  "idCardMasked": "440101********1234",
  "wechatBound": true
}
```

## 7. 迁移与初始化建议

### 7.1 当前环境说明

- 当前云端数据已清空时，可以直接按新模型初始化，无需再执行旧 `_openid` 清洗脚本

### 7.2 若未来导入旧数据

- 将旧 `users._openid` 迁移为 `user_identities(type=wechat_openid)`
- 将旧手机号主键关系补齐为 `userId`
- 清理旧的 `landlordOpenid`、`user1Openid`、`user2Openid` 等字段
- 迁移完成后强制所有用户重新登录

## 8. 数据清理建议

- 删除无效测试用户和测试房源
- 删除依赖旧字段结构的无效消息和会话记录
- 删除不再使用的 `_openid` 业务关联字段
- 保持 `user_identities` 与 `users.phone` 一致

## 9. 技术约束

- 使用微信云数据库作为唯一业务数据库
- 所有数据库操作仅允许在云函数中执行
- 页面禁止直接访问数据库
- 所有写操作需记录 `createTime` 与 `updateTime`
- 敏感字段必须在服务端完成加密、脱敏和权限校验
- 新功能不得重新引入 `phone` 或 `_openid` 作为主要业务关联键
