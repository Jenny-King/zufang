# 项目修复优先级清单

## P0：立即处理

### 1. 修复 bootstrap 云函数的生产环境保护
- 影响文件：
  - `cloudfunctions/bootstrap/index.js`
- 原因：
  - 当前仅依赖 `process.env.NODE_ENV === "production"` 判断生产环境，默认 `NODE_ENV` 未设置时保护失效。
  - 一旦请求体携带 `allowBootstrap: true`，初始化和清理逻辑可能在非预期环境执行。
- 目标：
  - 改为基于真实云环境标识判断是否允许执行。
  - 保证生产环境即使传入 `allowBootstrap: true` 也无法执行。

### 2. 为房源创建/编辑补齐服务端强校验
- 影响文件：
  - `cloudfunctions/house/index.js`
- 原因：
  - `create` 和 `update` 当前允许把必填字段写成空值、`0` 或非法类型，直接污染主业务数据。
- 目标：
  - 为 `title`、`price`、`address`、`type`、`area`、`paymentMethod`、`images`、`contactPhone` 等关键字段建立统一校验。
  - `update` 不允许把现有合法房源改坏。

### 3. 修复聊天链路对目标用户的缺失校验
- 影响文件：
  - `cloudfunctions/chat/index.js`
- 原因：
  - 当前可使用任意 `targetUserId` 创建会话，也可向无效 `receiverId` 写消息和通知。
  - 这会污染 `conversations`、`chat_messages`、`messages` 三个集合。
- 目标：
  - 创建会话前确认目标用户存在且有效。
  - 发消息前确认接收方有效且属于当前会话参与者。

### 4. 重新定义“实名认证”链路
- 影响文件：
  - `cloudfunctions/auth/index.js`
  - `package-auth/pages/verify/index.js`
- 原因：
  - 现在只有格式校验，仍会把 `verified` 直接置为 `true`，与功能语义不符。
- 目标：
  - 如果当前版本不接真实认证，至少要把状态命名、文案和流程改成“资料登记”而不是“实名认证”。
  - 如果保留实名认证语义，则必须接入真实校验或增加明确的审核状态。

### 5. 修复收藏/历史的孤儿数据写入
- 影响文件：
  - `cloudfunctions/favorite/index.js`
  - `cloudfunctions/history/index.js`
- 原因：
  - 当前可以对不存在或已删除房源写收藏、写历史。
  - 这会持续制造孤儿关系，并进一步导致分页总数和列表显示不一致。
- 目标：
  - 写入前必须校验房源存在且为 `active`。
  - 增加必要的数据清理策略，处理历史孤儿记录。

## P1：核心链路补齐

### 6. 补“忘记密码/重置密码”入口与服务端 action
- 影响文件：
  - `package-auth/pages/login/index.wxml`
  - `cloudfunctions/auth/index.js`
  - `docs/待办清单.md`
- 原因：
  - 文档已标记为 P1，但页面没有入口，云函数也没有 `resetPassword`。
- 目标：
  - 登录页增加重置密码入口。
  - `auth` 云函数补齐对应 action 和接口文档。

### 7. 补资料编辑页与通知页
- 影响文件：
  - `app.json`
  - `pages/profile/index.wxml`
  - `package-profile/pages/*`
- 原因：
  - 文档要求的 `package-profile/pages/edit-profile` 和 `package-profile/pages/notifications` 目前都不存在。
  - “我的”页面也没有资料编辑入口。
- 目标：
  - 生成页面文件、补路由、补入口。
  - 与当前个人中心流程联通。

### 8. 修复验证码链路仍依赖后台直接查库的问题
- 影响文件：
  - `cloudfunctions/auth/index.js`
  - `docs/待办清单.md`
- 原因：
  - `sendSmsCode()` 只是写 `sms_codes`，没有真实发送、限流、消费策略。
- 目标：
  - 明确这是开发态 mock 还是正式链路。
  - 若面向真实使用，至少补发送、限流、过期和重放防护。

### 9. 修复房源详情页配套设施英文展示
- 影响文件：
  - `package-house/pages/detail/index.js`
  - `package-house/pages/detail/index.wxml`
- 原因：
  - 页面当前展示 `elevator`、`parking` 等英文 key。
- 目标：
  - 建立设施 key 到中文标签的映射。
  - 确保详情页与发布页口径一致。

### 10. 修复收藏/历史分页总数不一致
- 影响文件：
  - `cloudfunctions/favorite/index.js`
  - `cloudfunctions/history/index.js`
- 原因：
  - 当前 `total` 来自原始关系表计数，但 `list` 会过滤掉失效房源。
- 目标：
  - 确保 `total` 与最终返回列表口径一致。
  - 避免前端出现“还有数据但显示为空”的分页错觉。

## P2：接口与架构一致性

### 11. 统一云函数响应结构
- 影响文件：
  - `cloudfunctions/auth/index.js`
  - `cloudfunctions/user/index.js`
  - `cloudfunctions/house/index.js`
  - `cloudfunctions/favorite/index.js`
  - `cloudfunctions/history/index.js`
  - `cloudfunctions/chat/index.js`
  - `cloudfunctions/map/index.js`
  - `cloudfunctions/bootstrap/index.js`
- 原因：
  - 当前成功和失败返回结构不完全稳定，未完全对齐文档中的 `{ code, data, message }`。
- 目标：
  - 所有云函数统一输出结构。
  - 避免前端和测试针对不同函数写分支兼容。

### 12. 收敛重复的鉴权辅助逻辑
- 影响文件：
  - `cloudfunctions/user/index.js`
  - `cloudfunctions/house/index.js`
  - `cloudfunctions/favorite/index.js`
  - `cloudfunctions/history/index.js`
  - `cloudfunctions/chat/index.js`
- 原因：
  - `getSessionByAccessToken()` 与 `resolveCurrentUser()` 在多个云函数中重复拷贝。
- 目标：
  - 提取共享鉴权模块，统一会话校验与当前用户解析逻辑。

### 13. 清理 service 层绕过统一调用链的问题
- 影响文件：
  - `services/house.service.js`
- 原因：
  - `uploadHouseImage()` 仍直接调用 `wx.cloud.uploadFile`。
- 目标：
  - 明确图片上传是否允许作为例外能力。
  - 若不允许，需补统一封装或改造架构文档。

### 14. 清理页面中的硬编码云函数名/路由
- 影响文件：
  - `pages/publish/edit.js`
  - 多个页面日志埋点
- 原因：
  - 当前仍有硬编码路由与 `house.create`、`auth.logout` 一类字符串埋点。
- 目标：
  - 路由走配置常量。
  - 日志口径明确是否允许使用接口标识字符串；若不允许，需要统一封装。

## P3：测试、文档与遗留清理

### 15. 为 chat/favorite/history/house 补关键业务测试
- 影响文件：
  - `cloudfunctions/chat/__tests__/index.test.js`
  - `cloudfunctions/favorite/__tests__/index.test.js`
  - `cloudfunctions/history/__tests__/index.test.js`
  - `cloudfunctions/house/__tests__/index.test.js`
- 原因：
  - 目前大多只有 smoke 测试，无法兜住权限、写入校验和分页逻辑。
- 目标：
  - 至少覆盖成功路径、参数错误、越权访问、关键写库副作用。

### 16. 删除或处理死代码 Behavior
- 影响文件：
  - `behaviors/auth-required.js`
- 原因：
  - 已未被页面使用，维护成本高于价值。
- 目标：
  - 要么正式接入，要么删除并更新文档。

### 17. 处理废弃脚本与模板残留
- 影响文件：
  - `uploadCloudFunction.sh`
- 原因：
  - 当前是无法执行的模板残留，容易误导使用者。
- 目标：
  - 删除、替换或在文档中明确废弃。

### 18. 更新文档中的实际结构与脚本说明
- 影响文件：
  - `docs/技术架构设计.md`
  - `docs/AGENTS.md`
  - `README.md`
- 原因：
  - 当前仓库已有 `scripts/`、联调脚本、部署脚本，但文档记录不完整。
  - `README.md` 内容与项目实际不符。
- 目标：
  - 让新接手的人能从文档直接理解目录、脚本用途和当前状态。

## 推荐修复顺序

1. 先做 P0，避免继续产生脏数据和安全暴露。
2. 再做 P1，补齐登录、资料和通知等核心业务缺口。
3. 然后做 P2，统一接口口径并收敛重复逻辑。
4. 最后做 P3，补测试、清死代码、补文档。
